# AI Agent Instructions for Homelab API

## 1. Project Context & Role
You are an expert Go backend engineer. You are working on a lightweight, highly efficient "Modular Monolith" API built in Go. 
Your primary goal is to maintain a low memory footprint, use the standard library wherever possible, and prevent tight coupling between business domains.

## 2. Tech Stack Constraints
Do NOT introduce third-party frameworks unless explicitly instructed.
*   **Routing:** Go 1.22+ standard library `net/http` (`http.ServeMux`). **DO NOT** use Gin, Fiber, Echo, or Chi.
*   **Database:** PostgreSQL using `github.com/jackc/pgx/v5` and `sqlc`. **DO NOT** use GORM, Ent, or database/sql directly.
*   **Redis:** `github.com/redis/go-redis/v9`.
*   **AWS / S3:** `github.com/aws/aws-sdk-go-v2` (Config, S3, and Manager).
*   **Image Processing:** Standard `image` package and `github.com/disintegration/imaging`.

## 3. Architectural Rules (Modular Monolith)
The project relies on strict domain boundaries located inside the `internal/` directory.

### DO:
*   Keep domains strictly isolated (e.g., `internal/users`, `internal/catalog`).
*   Communicate between domains using Go interfaces, injected via `cmd/api/main.go`.
*   Keep all dependency injection and cross-domain wiring inside `cmd/api/main.go`.
*   Place cross-cutting concerns (logger, database connections, auth middleware) in `internal/shared`.

### DON'T:
*   Do not query one domain's database tables from another domain. No cross-domain SQL joins.
*   Do not create global `controllers`, `models`, or `repositories` folders at the root level. Group by feature/domain.

## 4. Go Language Best Practices
Write idiomatic, modern Go code.

### DO:
*   **Pass Context First:** Always pass `ctx context.Context` as the first parameter to functions dealing with databases, network calls, or external APIs to handle timeouts and cancellations correctly.
*   **Accept Interfaces, Return Structs:** Functions should accept interfaces for their dependencies but return concrete structs.
*   **Return Early:** Use guard clauses to handle errors or edge cases immediately to avoid deep nesting.
*   **Wrap Errors:** When returning errors from lower levels (like DB calls), wrap them with context using `fmt.Errorf("doing something: %w", err)`.
*   **Use Value Semantics:** Pass structs by value by default. Only pass by pointer (`*Struct`) if you need to mutate the struct or if it is exceptionally large.

### DON'T:
*   Do not use `panic()` for standard error handling. Return `error` as the last return value.
*   Do not use global variables for database connection pools or external clients. Inject them as dependencies.
*   Do not ignore errors (e.g., never use `_ = someFuncThatReturnsError()`).

## 5. Database & sqlc Workflow
*   Database schema changes must be written in raw SQL inside `internal/<domain>/schema.sql`.
*   Queries must be written in raw SQL inside `internal/<domain>/queries.sql`.
*   Always run `sqlc generate` to create the corresponding Go code; never manually edit the generated `*.sql.go` files.

## 7. Separation of Concerns (The 3-Tier Domain)

Within each domain (e.g., `internal/users`), strictly separate concerns into three layers. AI agents must not leak logic between these layers.

* **Handler Layer (`handler.go`):** Deals ONLY with HTTP. It parses `http.Request`, reads URL parameters, decodes JSON bodies, passes data to the Service layer, and formats the JSON `http.ResponseWriter`. **No business rules or database calls belong here.**
* **Service Layer (`service.go`):** The brain of the domain. It enforces business rules, processes data, and orchestrates calls to the Repository. **It knows nothing about HTTP or JSON.**
* **Repository Layer (`repository.go`):** Deals ONLY with data persistence. It wraps the `sqlc`-generated code or AWS S3 clients. **It contains no business logic.**

## 8. Interface Placement (The Go Way)

Unlike C# or Java where interfaces are defined next to the implementation, Go uses implicit interfaces.

### DO:

* **Define interfaces where they are consumed, not where they are produced.**
* If the `Handler` needs the `Service`, define the interface inside `handler.go`.
* If the `Service` needs the `Repository`, define the `Repository` interface inside `service.go`.
* If the `catalog` domain needs to check a user's status, `internal/catalog/service.go` should define a `UserChecker` interface. `cmd/api/main.go` will pass the `users.Service` into it.

### DON'T:

* Do not create an `interfaces/` folder.
* Do not define an interface just to have one. If a struct only has one implementation and isn't being mocked for tests yet, just use the concrete struct.

## 9. Environment Variables

* **Centralized Loading:** Load all environment variables exactly once at application startup in `cmd/api/main.go` (or via `internal/shared/config`).
* **Dependency Injection:** Pass specific config values down to the domains as simple variables or structs (e.g., pass the `JWTSecret` string directly to the auth service).
* **No Hidden State:** **NEVER** use `os.Getenv()` inside `internal/users/` or deeply nested business logic. It creates hidden dependencies and makes the code impossible to test.

## 10. Constants and Enums

* **Domain-Specific:** If a constant only belongs to one feature (e.g., `const MaxUploadSize`, `const StatusActive`), define it inside that domain's `models.go` or `service.go`.
* **App-Wide:** If a constant is truly global (e.g., HTTP header keys, context keys for middleware), place it in `internal/shared/constants/`.
* **No Global Constant Dumps:** Do not create a single massive `constants.go` file at the root of the project containing unrelated values.
* **Go Enums:** Use custom types and `iota` for enums (e.g., `type UserRole string`, followed by `const ( RoleAdmin UserRole = "ADMIN" ... )`).
